Taproot coinbase

P2Pool scientific basics

A Taproot output (bc1p…, Bech32m) pays via OP_1 <taproot key>, with an empty scriptSig — validation lives in the witness. A single Taproot address can encode many spending conditions through MAST, so a coinbase could, in principle, distribute a block reward to many parties under one output: a key-path spend for the straightforward case, and hidden script-paths for conditional distributions (e.g. by time, participation, or contribution).

Using it for pool payouts

A pool can generate a Taproot address combining members’ keys and scripts; the coinbase pays that address; rewards mature after 100 blocks; distribution then follows the key-path (cooperative) or a script-path (conditional / dispute). Members can verify the agreed distribution from the public key and committed script hashes without knowing the private key (public-key transparency + MAST pre-commitment). MuSig lets the key be co-generated so no single party holds it.

Where it helps — and where it does not

Taproot can consolidate many payout conditions under one address and improve privacy/efficiency — useful against hardware that caps the number of coinbase addresses. But it addresses the symptom, not the cause: firmware address-limits and manufacturer policy still bind; reused-address and pool-managed-address workarounds remain operational, not structural.

V37How V37 actually solves “many miners, one coinbase.”

This page explores a scripting answer: pack many payout conditions into one Taproot output, often with a primary key “possibly held by the pool operator.” That key-path-with-an-operator-key is a custodial shape — exactly what a decentralized pool must avoid — and MAST consolidation still cannot pay an unbounded miner set within a bounded coinbase.

V37 takes a different route and needs no operator key and no covenant: it separates accounting from payment. Every miner’s owed balance lives in a committed, per-node ledger at no coinbase cost; each found block’s coinbase pays the largest owed balances, in canonical order, up to a fixed budget; the rest carry forward and are paid in a later block. Unbounded miners, bounded coinbase, deterministic, nobody’s work lost — and every payout is an ordinary output the whole network re-derives, so nothing can be skimmed. See the Purple Paper, §9 (Paying Many from a Small Coinbase) and §7 (Many Chains).

Taproot remains a useful primitive (each output can still be a Taproot key for privacy/efficiency); it is simply not, by itself, the answer to the coinbase-scaling problem.
