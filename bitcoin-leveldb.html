Bitcoin LevelDB

P2Pool scientific basics — reading block data directly from disk

Bitcoind since 0.8 maintains two databases, the block index (in $DATADIR/blocks/index) and the chainstate (in $DATADIR/chainstate). The block index maintains information for every block, and where it is stored on disk. The chain state maintains information about the resulting state of validation as a result of the currently best known chain.

Block index key/value pairs

Inside the block index, the used key/value pairs are:

  • 'b' + 32-byte block hash → block index record. Each record stores: the block header; the height; the number of transactions; to what extent this block is validated; in which file, and where in that file, the block data is stored; in which file, and where in that file, the undo data is stored.
  • 'f' + 4-byte file number → file information record. Each record stores: the number of blocks stored in the block file with that number; the size of the block file ($DATADIR/blocks/blkNNNNN.dat); the size of the undo file (revNNNNN.dat); the lowest and highest height of blocks stored in that file; the lowest and highest timestamp of blocks stored in that file.
  • 'l' → 4-byte file number: the last block file number used.
  • 'R' → 1-byte boolean ('1' if true): whether we're in the process of reindexing.
  • 'F' + 1-byte flag name length + flag name string → 1-byte boolean: various flags. Currently defined flags include 'txindex' (whether the transaction index is enabled).
  • 't' + 32-byte transaction hash → transaction index record (optional, only if 'txindex' is enabled). Each record stores: which block file number the transaction is stored in; which offset into that file the block is stored at; the offset from the start of that block to where the transaction itself is stored.

Chain state key/value pairs

  • 'C' + 32-byte transaction hash + output index length + output index (v0.15 onwards) → a single UTXO record: whether the transaction was a coinbase; which height block contains it; the scriptPubKey and amount for this unspent output.
  • 'c' + 32-byte transaction hash (pre-v0.14) → the UTXO record for that transaction (all UTXOs from a single transaction; only present if at least one unspent output remains): the transaction version; whether coinbase; which height block contains it; which outputs are unspent; the scriptPubKey and amount for those unspent outputs.
  • 'B' → 32-byte block hash: the block hash up to which the database represents the unspent transaction outputs.

The latest version of bitcoind uses obfuscation of the value in key/value pairs — you need to XOR with the obfuscation key to get the real value. I won't go into the specific serialization details of the particular records; they're often specially designed to be compact on disk, and not really intended to be easily usable by other applications (LevelDB doesn't support concurrent access from multiple applications anyway). There are several RPC methods for querying data from the databases (getblock, gettxoutsetinfo, gettxout) without needing direct access.

As you can see, only headers are stored inside this database. The actual blocks and transactions are stored in the block files, which are not databases, but just raw append-only files that contain the blocks in network format. If you just want to access some records, iterate over the keys and stop when you've read enough.

Parsing a block-index record and its header

import binascii 
BLOCK_HAVE_DATA = 8 #!< full block available in blk*.dat 
BLOCK_HAVE_UNDO = 16 #!< undo data available in rev*.dat 

def encode_varint(number):
    """Encodes a non-negative integer using the MSB base-128 scheme."""
    if number < 0:
        raise ValueError("Only non-negative integers can be encoded.")
    result = []
    while True:
        byte = number & 0x7F          # Extract lower 7 bits
        number >>= 7                  # Shift right by 7
        if number:
            byte |= 0x80              # Set high bit for continuation
        result.append(byte)
        if number == 0:
            break
    return bytes(result)

def decode_varint(stream):
    """Decodes a variable-length integer from the MSB base-128 format."""
    n = 0
    while True:
        chData = ord(stream.get(1))
        n = (n << 7) | (chData & 0x7F)
        if chData & 0x80:
            n += 1
        else:
            return n

def read_int(stream, bits):
    data = stream.get(bits//8)
    data.reverse()
    return binascii.b2a_hex(data) if bits > 64 else int(binascii.b2a_hex(data), 16)

class Stream:
    """Class to handle byte stream"""
    def __init__(self, hexdata):
        self.data = bytearray(bytes.fromhex(hexdata))
        self.data.reverse()
    def get(self, n):
        result = self.data[:n]
        self.data = self.data[n:]
        return result

class BlockHeader:
    def __init__(self, stream):
        self.nVersion       = read_int(stream, 32)
        self.hashPrev       = read_int(stream, 256)
        self.hashMerkleRoot = read_int(stream, 256)
        self.nTime          = read_int(stream, 32)
        self.nBits          = read_int(stream, 32)
        self.nNonce         = read_int(stream, 32)

class VarintCBlockIndex:
    def __init__(self, stream):
        self.nVer     = decode_varint(stream)
        self.nHeight  = decode_varint(stream)
        self.nStatus  = decode_varint(stream)
        self.nTx      = decode_varint(stream)
        self.nFile    = decode_varint(stream) if self.nStatus & (BLOCK_HAVE_DATA | BLOCK_HAVE_UNDO) else -1
        self.nDataPos = decode_varint(stream) if self.nStatus & BLOCK_HAVE_DATA else -1
        self.nUndoPos = decode_varint(stream) if self.nStatus & BLOCK_HAVE_UNDO else -1

Key structure (b + 32-byte block hash)

The key for each block index record begins with the letter 'b' to distinguish it from other types of entries (e.g. transaction index entries start with 't'). Following the 'b' is the 32-byte (256-bit) hash of the block, a unique identifier. The block hash is typically represented in little-endian byte order in the database.

Value structure (block index record)

Each block index record contains a combination of data:

  • Block Header — 80 bytes total: Version (4) · Previous Block Hash (32) · Merkle Root (32) · Timestamp (4) · Bits (difficulty target, 4) · Nonce (4).
  • Height — a varint giving the block's height (how many blocks precede it).
  • Number of Transactions — a varint for the total transactions in the block.
  • Validation Status — a varint of flags: whether block data is fully available (BLOCK_HAVE_DATA), whether undo data is available (BLOCK_HAVE_UNDO).
  • File Location and Position — if block data is available: a varint file number (blkXXXXX.dat) and a varint byte offset where the block data starts; if undo data is available: a varint file number (revXXXXX.dat) and a varint byte offset.

Varints: Bitcoin uses varints for space efficiency — smaller numbers use fewer bytes, so the record size is variable. Data availability: not all block/undo data is in the LevelDB index; the validation-status flags indicate whether the data is present and where to find it in the block files. Endianness: the block hash and the header fields are stored little-endian.

Example (simplified). A record looks like:

Key:   b<block_hash>
Value: <block_header_bytes><varint_height><varint_tx_count><varint_status>
       <varint_file_num><varint_data_pos><varint_undo_file_num><varint_undo_pos>

This record tells you: the block hash; the block header data; the block height; the number of transactions; the block's validation status; and where to find the block data and undo data on disk (if available).

V37This method, in practice.

The "only headers in the database; the blocks themselves are raw append-only files" point is exactly what makes direct, daemon-free reads possible. The Real difficulty chart on this site was sourced precisely this way: rather than query a running node over RPC, we read blocks/blk00000.dat straight off a full node's disk and parsed the 80-byte headers sequentially (magic → size → header → nBits + double-SHA-256 block hash) for the first 4,032 blocks. No RPC, no rate limit, no daemon — just the raw files this page describes. The same property underpins the V37 work: a node can reconstruct committed state from compact, on-disk records and prove it with log-size Merkle proofs — see the Purple Paper, §9 (Verifying Payment).
